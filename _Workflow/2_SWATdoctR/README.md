# Mini_setup_CREATE / 2_SWATdoctR

## SWATdoctR verification workflow manual

This README documents the `2_SWATdoctR` part of the `Mini_setup_CREATE` repository. It is intended for workshop teaching and for repository documentation.

## Scope

This folder does **not** build the SWAT+ model. It verifies an already prepared SWAT+ TxtInOut-style setup stored in `clean_setup`.

The main purpose is quality assurance:

- run diagnostic SWAT+ simulations;
- extract selected model outputs;
- generate diagnostic plots;
- identify potential problems in climate inputs, water balance, management operations, plant growth, plant stress, point sources, or tile drainage.

This README does not document the earlier model-building workflow in `1_Setup`, except where needed to explain the origin of `clean_setup`.

## Repository folder overview

The `2_SWATdoctR` folder is compact. It contains an RStudio project file, one workflow script, and a complete `clean_setup` folder containing SWAT+ input files.

The `clean_setup` folder includes standard SWAT+ input files such as climate files, operation files, HRU files, routing/connectivity files, basin files, plant/soil files, reservoir files, and decision-table files.

| Path | Meaning |
|---|---|
| `2_SWATdoctR/DoctR.Rproj` | RStudio project file. It sets UTF-8 encoding, two spaces for tabs, and standard RStudio project behavior. |
| `2_SWATdoctR/workflow.R` | Main SWATdoctR script. It runs three verification simulations and generates `plots.pdf`. |
| `2_SWATdoctR/clean_setup/` | SWAT+ model input folder used by `run_swat_verification()`. It must contain all required model input files and the SWAT+ executable expected by the package/workflow. |
| `2_SWATdoctR/plots.pdf` | Expected output generated by `workflow.R` after all diagnostic plotting commands complete. This file is not an input; it is created by the script. |

## Purpose of the SWATdoctR workflow

SWATdoctR is used here as a model setup verification tool. The goal is **not calibration**. The goal is to check whether the existing SWAT+ setup behaves plausibly before later calibration, validation, or scenario simulations.

The verification workflow checks the following areas.

| Verification area | Main question answered by the workflow |
|---|---|
| Climate and weather inputs | Are precipitation, temperature, snow, and annual climate patterns interpreted plausibly by the model? |
| Water balance | Do the water-balance components look physically plausible after the model run? |
| Management operations | Were scheduled agricultural operations actually triggered as expected? |
| Plant growth without stress | Can crops reach plausible maturity, biomass, and yield when stresses are removed? |
| Plant stress with stress modes active | Are low yields caused by water, aeration, temperature, nitrogen, or phosphorus limitations? |
| Point sources and tile drainage | If implemented, are point inputs and tile-flow outputs plausible? |

## Required software and packages

| Requirement | Why it is needed |
|---|---|
| R and RStudio | R runs `workflow.R`; the RStudio project file supports workshop execution. |
| `SWATdoctR` | Provides `run_swat_verification()` and the diagnostic plotting functions used by `workflow.R`. |
| `grid` | Used directly for `grid.newpage()`, `viewport()`, `pushViewport()`, and `popViewport()` when arranging plots in the PDF device. |
| `gridExtra` | Loaded by `workflow.R`. The inspected script does not explicitly call `gridExtra` functions, but it may support grid-based plot handling or earlier workflow variants. |
| SWAT+ executable and valid `clean_setup` files | `run_swat_verification()` must be able to run the SWAT+ model from the `clean_setup` folder. |

## Step-by-step explanation of `workflow.R`

### Step 1 — Load packages

```r
library(SWATdoctR)
library(grid)
library(gridExtra)
```

The script loads `SWATdoctR` for verification runs and plots. It also loads `grid` and `gridExtra` for multi-page PDF plotting. `grid` functions are used explicitly when creating a fresh page, defining a viewport, printing a plot into that viewport, and closing the viewport.

### Step 2 — Define three verification simulations

The script performs three separate runs of the same `clean_setup` model. Each call uses the same project path, output groups, date range, warm-up skip, and folder-cleaning behavior. The only intended difference is the `nostress` setting.

| Run object | Script setting | Project path | Outputs read | Date range in shown code | Warm-up / skip | Folder behavior |
|---|---|---|---|---|---|---|
| `Run_1` | `nostress = 0` | `./clean_setup` | `wb`, `mgt`, `plt` | 2004-01-01 to 2023-12-31 | `years_skip = 3` | `keep_folder = FALSE` |
| `Run_2` | `nostress = 1` | `./clean_setup` | `wb`, `mgt`, `plt` | 2004-01-01 to 2023-12-31 | `years_skip = 3` | `keep_folder = FALSE` |
| `Run_3` | `nostress = 2` | `./clean_setup` | `wb`, `mgt`, `plt` | 2004-01-01 to 2023-12-31 | `years_skip = 3` | `keep_folder = FALSE` |

Example call for `Run_1`:

```r
Run_1 <- run_swat_verification(
  "./clean_setup",
  outputs = c("wb", "mgt", "plt"),
  start_date = 20040101,
  end_date = 20231231,
  years_skip = 3,
  nostress = 0,
  keep_folder = FALSE
)
```

The first three years are skipped from written outputs, so diagnostics emphasize the post-warm-up period.

> **Verification note:** the uploaded manual states in one place that `clean_setup/time.sim` is configured with `yrc_start = 2004` and `yrc_end = 2024`, but the shown R code uses `end_date = 20231231`. Before running the workflow, verify whether the intended final diagnostic year is 2023 or 2024.

### Step 3 — Combine the three run objects

```r
sim_list <- list(all_str = Run_1, no_str = Run_2, nutrient_str = Run_3)
```

The combined list allows SWATdoctR plotting functions to compare the same variable across different stress settings. This is used later for yield and PHU comparison plots.

For teaching, the label `nutrient_str` should be reviewed because it may not clearly communicate what `Run_3` represents in the installed SWATdoctR/SWAT+ version.

### Step 4 — Open a multi-page PDF device

```r
pdf("plots.pdf", width = 15, height = 25)
```

All subsequent plots are written to a single PDF file named `plots.pdf`. The dimensions are large: 15 inches wide and 25 inches high. This is suitable for complex plots, but the resulting PDF pages may be large.

For a workshop, it may be useful to create smaller separate files or save individual PNG files as well.

### Step 5 — Plot annual climate diagnostics

```r
grid.newpage()
pushViewport(viewport(width = 0.8, height = 1))
print(plot_climate_annual(Run_1), newpage = FALSE)
popViewport()
```

This plot checks the annual climate variables from `Run_1`. It is intended to reveal whether climate inputs were read and interpreted plausibly.

Examples of problems to look for:

- missing years;
- unrealistic precipitation totals;
- implausible temperatures;
- sudden discontinuities caused by input formatting problems.

### Step 6 — Plot monthly snow diagnostics

```r
grid.newpage()
pushViewport(viewport(width = 0.8, height = 0.3))
print(plot_monthly_snow(Run_1), newpage = FALSE)
popViewport()
```

The snow diagnostic focuses on monthly snow behavior. This is useful in cold or mixed climates because unrealistic snow accumulation or melt timing can indicate problems in climate inputs, temperature thresholds, elevation effects, or snow-related model settings.

### Step 7 — Plot simplified and detailed water balance

```r
print(plot_waterbalance(Run_1, simplified = TRUE), newpage = FALSE)
print(plot_waterbalance(Run_1), newpage = FALSE)
```

Two water-balance plots are generated. The simplified version is for quick screening, while the full version gives more detail.

These plots should be checked for:

- implausible magnitudes;
- missing components;
- unrealistic dominance of one term;
- symptoms of disconnected routing.

### Step 8 — Plot yield and PHU at harvest/kill for `Run_1`

```r
plot_variable_at_harvkill(Run_1, variable = 'yield')
plot_variable_at_harvkill(Run_1, variable = 'phu')
```

These plots summarize crop yield and potential heat-unit fraction at harvest/kill events. PHU values close to crop maturity are important for checking whether growing periods and crop parameters are compatible with the climate and management schedule.

Investigate cases with:

- very low yields;
- extreme yields;
- PHU values far below maturity;
- very large PHU values.

### Step 9 — Compare yield and PHU across stress settings

```r
plot_variable_at_harvkill(sim_list, variable = 'yield')
plot_variable_at_harvkill(sim_list, variable = 'phu')
```

These plots compare `Run_1`, `Run_2`, and `Run_3`. Their role is to show how stress settings affect yield and crop maturity indicators.

For example, a large difference between no-stress and stress-active runs can indicate that stress factors, fertilizer inputs, irrigation assumptions, crop parameters, or soil properties need further inspection.

### Step 10 — Plot stress indicators for `Run_1`

```r
plot_variable_at_harvkill(Run_1, variable = 'stress')
```

This plot checks stress values at harvest/kill events when all stresses are active. It should help identify whether low yield is linked to water stress, aeration stress, temperature stress, nitrogen stress, or phosphorus stress, depending on the variables made available by SWATdoctR.

### Step 11 — Optional point source plot

```r
# plot_ps(Run_1)
```

The point-source plot is commented out in the repository script. It should only be enabled if the model contains point-source inputs and the relevant outputs were extracted.

The official SWATdoctR workflow recommends checking point sources and tile drainage if these processes are implemented.

### Step 12 — Plot daily HRU plant variables for selected HRUs

```r
plot_hru_pw_day(
  Run_1,
  hru_id = c(103, 117, 160, 205, 274),
  var = c('lai', 'bioms'),
  years = 2010:2023,
  add_crop = TRUE
)
```

This is a detailed diagnostic for selected HRUs. It plots daily plant-water/plant-growth variables, specifically LAI and biomass, for HRUs `103`, `117`, `160`, `205`, and `274`.

With `years = 2010:2023`, the plot covers 2010 through 2023. `add_crop = TRUE` overlays crop information, which helps link growth behavior to the active crop and management schedule.

### Step 13 — Close the PDF

```r
dev.off()
```

This closes the PDF graphics device. If this line is not executed because of an error in an earlier plot, `plots.pdf` may be incomplete or locked by the R session.

In such a case, run `dev.off()` manually until no PDF device remains open, then rerun the failed section.

## Expected outputs

| Output | Created by | Content / use |
|---|---|---|
| `Run_1` | `run_swat_verification()` | In-memory list of extracted SWAT+ outputs for all stress factors active, according to the `workflow.R` comments. |
| `Run_2` | `run_swat_verification()` | In-memory list of extracted SWAT+ outputs for no-stress verification. |
| `Run_3` | `run_swat_verification()` | In-memory list of extracted SWAT+ outputs for the third stress configuration. Verify exact semantics of `nostress = 2` for the installed SWATdoctR version. |
| `sim_list` | Base R `list()` | Combined object used for cross-run yield and PHU comparisons. |
| `plots.pdf` | `pdf()` graphics device | Multi-page diagnostic PDF containing climate, snow, water-balance, harvest/kill, stress, and HRU daily plots. |
| Temporary verification run folder | SWATdoctR internals | Normally deleted because `keep_folder = FALSE`. Set `keep_folder = TRUE` only when debugging run files. |

## How to interpret the generated plots

| Plot | What to check | Possible issue if abnormal |
|---|---|---|
| `plot_climate_annual(Run_1)` | Annual precipitation, temperature, and other climate summaries. | Wrong units, missing years, incorrect station assignment, or corrupted weather input. |
| `plot_monthly_snow(Run_1)` | Seasonal snow accumulation and melt behavior. | Temperature/snow parameter inconsistency, implausible snowfall/melt, or elevation-related mismatch. |
| `plot_waterbalance(Run_1, simplified = TRUE)` | First-pass water-balance plausibility. | Disconnected hydrology, unrealistic evapotranspiration, excessive runoff, or missing outputs. |
| `plot_waterbalance(Run_1)` | Detailed water-balance components. | More specific diagnosis of the same water-balance problems. |
| `plot_variable_at_harvkill(..., 'yield')` | Yield at crop harvest/kill events. | Management schedule, crop parameters, stress limitation, or harvest operation problem. |
| `plot_variable_at_harvkill(..., 'phu')` | Potential heat-unit fraction at harvest/kill. | Growing season too short/long, incorrect PHU, incorrect planting/harvest timing, or crop mismatch. |
| `plot_variable_at_harvkill(..., 'stress')` | Stress factors at harvest/kill. | Water, aeration, temperature, nitrogen, or phosphorus limitation may be dominating. |
| `plot_hru_pw_day()` | Daily LAI and biomass for selected HRUs. | Crop development anomaly, wrong crop transition, failed planting/harvest, or unrealistic biomass accumulation. |

## Workshop execution notes

1. Open `2_SWATdoctR/DoctR.Rproj` in RStudio.
2. Confirm that `clean_setup/` contains the complete SWAT+ setup and the required executable.
3. Install/load the required packages before the workshop.
4. Run `workflow.R` section by section for teaching, or source it as a complete workflow after checking package availability.
5. Review `plots.pdf` after the run completes.
6. If a plotting or model-run error interrupts the script, close open PDF devices with `dev.off()` before rerunning.
7. Use `keep_folder = TRUE` temporarily when debugging SWAT+ run files.

## Important checks before using this folder in a workshop

- Verify the installed SWATdoctR version and the meaning of `nostress = 0`, `nostress = 1`, and `nostress = 2`.
- Verify whether the intended final diagnostic year is 2023 or 2024.
- Confirm that `clean_setup/time.sim` is consistent with the `start_date` and `end_date` used in `workflow.R`.
- Confirm that the selected HRU IDs in `plot_hru_pw_day()` exist in the setup.
- Enable `plot_ps(Run_1)` only if point-source inputs and outputs are available.
- Avoid modifying `clean_setup` during the workshop unless the goal is to demonstrate troubleshooting.
